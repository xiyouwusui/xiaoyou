# Omnibot App Update Worker

Cloudflare Worker for public app update checks, authenticated release metadata management, APK delivery through Cloudflare R2, a built-in admin console, and usage statistics via Workers Analytics Engine. Release metadata and APK files are stored in R2; no KV namespace is required.

## Admin console

`GET /admin` serves a self-contained management console (token login via `ADMIN_TOKEN`):

- **版本管理** — create/edit releases, write the changelog shown in the app's update card, toggle stable/beta track and draft state, upload APKs straight from the browser (large files use R2 multipart upload automatically, SHA-256 is computed client-side), copy download links, delete releases/assets.
- **数据统计** — daily update-check/download trends, unique devices, device model / app version / Android version distributions, per-asset download counts. Backed by Workers Analytics Engine.

A changelog curated in the console survives CI republishes: the CI payload carries no `releaseNotes`, and the worker keeps existing notes when an upsert omits them.

## Routes

- `GET /updates?currentVersion=0.5.0.3&edition=standard&source=worker&includeBeta=true`
  - Public endpoint used by the Android app.
  - Optional anonymous stats params recorded to Analytics Engine:
    `deviceBrand`, `deviceModel`, `osVersion`, `sdkInt`, `installId` (random UUID generated by the app).
  - Returns `apkDownloadUrl` pointing at this Worker.
- `GET /downloads/:tag/:asset`
  - Public APK download endpoint backed by R2. APK downloads are counted in Analytics Engine.
- `GET /admin`
  - Admin console (public HTML shell; every API call requires the token).
- `GET /admin/api/session`
  - Token check used by the console login.
- `GET /admin/releases` · `GET /admin/releases/:tag`
  - Requires `Authorization: Bearer <ADMIN_TOKEN>`.
- `POST /admin/releases`
  - Upserts a release. Empty/absent `releaseNotes`/`releaseUrl` keep the stored values; absent `assets` keeps stored assets (an explicit `[]` clears them).
- `PATCH /admin/releases/:tag`
  - Partial update (any of `version`, `track`, `draft`, `prerelease`, `publishedAt`, `releaseUrl`, `releaseNotes`, `assets`).
- `DELETE /admin/releases/:tag`
  - Removes a tag so clients stop seeing a retracted package.
- `PUT /admin/releases/:tag/assets/:asset`
  - Streams a small APK or `.apk.sha256` file into the bound R2 bucket.
- `DELETE /admin/releases/:tag/assets/:asset`
  - Deletes the R2 object and detaches it from the release metadata.
- `POST /admin/releases/:tag/assets/:asset?action=mpu-create`
- `PUT /admin/releases/:tag/assets/:asset?action=mpu-uploadpart&uploadId=...&partNumber=...`
- `POST /admin/releases/:tag/assets/:asset?action=mpu-complete&uploadId=...`
- `DELETE /admin/releases/:tag/assets/:asset?action=mpu-abort&uploadId=...`
  - Multipart upload flow for large APKs, keeping each Worker request below Cloudflare's request body limit.
- `GET /admin/analytics/:metric?days=30&limit=10`
  - Metrics: `summary`, `daily`, `devices`, `versions`, `os`, `downloads`.
  - Proxies preset SQL to the Analytics Engine SQL API (requires `CF_ACCOUNT_ID` + `CF_ANALYTICS_API_TOKEN`).

## Analytics Engine schema

One dataset (default `omnibot_app_updates`), event type in `index1`/`blob1`:

| Field | check event | download event |
|---|---|---|
| `blob2` | app currentVersion | — |
| `blob3` | latest offered version | release tag |
| `blob4` | edition | asset file name |
| `blob5`–`blob8` | device brand / model / OS version / SDK int | — |
| `blob9` | anonymous install id | — |
| `blob10` | request country | request country |
| `blob11`/`blob12` | track / source | — |
| `double1` | hasUpdate (1/0) | 0 |

Counts must be computed as `sum(_sample_interval)` (the admin console does this). Data retention is ~90 days.

## Deploy

Create an R2 bucket, bind it and the Analytics Engine dataset to the Worker, then configure the tokens.

```bash
wrangler r2 bucket create omnibot-app-updates
cp wrangler.toml.example wrangler.toml
# Put the created bucket name and your Cloudflare account id in wrangler.toml.
wrangler secret put ADMIN_TOKEN            # admin console + release API token
wrangler secret put CF_ANALYTICS_API_TOKEN # API token with "Account Analytics: Read"
wrangler deploy
```

Dashboard bindings (when not using wrangler):

- `R2 bucket` → variable `APP_UPDATE_BUCKET`
- `Analytics Engine` → variable `UPDATE_ANALYTICS`, dataset `omnibot_app_updates`

Notes:

- Without the Analytics Engine binding everything still works; stats are simply not recorded.
- Without `CF_ACCOUNT_ID`/`CF_ANALYTICS_API_TOKEN` the console's stats page shows a configuration hint; release management is unaffected.

Use the deployed Worker URL as:

- Android Gradle property: `OMNIBOT_UPDATE_WORKER_URL`
- GitHub Actions secret: `APP_UPDATE_WORKER_URL`

Use the same token as the GitHub Actions secret `APP_UPDATE_WORKER_TOKEN`.

GitHub release publishing uploads staged APKs with the helper script. Large APKs use Worker-backed R2 multipart upload automatically:

```bash
python3 scripts/upload_release_asset_to_worker.py \
  --worker-url "$APP_UPDATE_WORKER_URL" \
  --token "$APP_UPDATE_WORKER_TOKEN" \
  --tag v1.6.2 \
  --file OpenOmniBot-v1.6.2-standard.apk \
  --content-type application/vnd.android.package-archive \
  --sha256 "$APK_SHA256"
```

CI publishes the release tag without `releaseNotes`; write the user-facing changelog in the admin console (it is what the app's update card displays).
